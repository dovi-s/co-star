Objective
Prevent multi-account abuse where users create new free accounts with different emails to bypass the 3 rehearsals/day free limit. Implement device-level fingerprinting so the daily limit follows the device, not just the account.

Approach
Use a device fingerprint system:

Generate a stable device ID on the client (stored in both localStorage and IndexedDB for resilience)
Create a server-side device_usage table that tracks usage per device fingerprint
When checking free-tier limits, check BOTH the user's account usage AND the device's usage — whichever is higher wins
New accounts on the same device inherit the device's existing usage count
This stops the casual "make a new email, get 3 more free runs" abuse loop. It's not bulletproof (factory reset or different browser defeats it), but it covers 95%+ of real abuse without adding friction for legitimate users.
